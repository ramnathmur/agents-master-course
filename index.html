📓 Agents Master Course

Stop reading about agents. Start architecting them.

A 14-chapter, AI-first, hands-on course on the Claude Agent SDK. Every chapter is a Markdown script that Claude Desktop runs with you — adopting a named persona, asking Socratic questions, adapting to how you answer, and grading you at the end. You leave each chapter with a runnable Python file in IntelliJ and a cumulative HTML brief of everything you've learned.

📦 14 chapters · 18 Python files ⏱ ~16 hours of deep work 💳 Runs on Claude Max · no API key 🐍 Python · IntelliJ / PyCharm

🎯 What this course is

The Claude Agent SDK gives you the primitives to build agents — query, ClaudeSDKClient, @tool, hooks, sub-agents, permission modes. Most resources teach you to recognize these primitives. This course teaches you to architect with them.

The spine is twelve progressive projects, plus a Chapter 0 setup, a Chapter 12.5 "Cold Build" stress test, and a Chapter 13 on production surfaces. Each project adds exactly one architectural concept to the one before. Chapter 1 is a single LLM call — no agent at all. Chapter 12 is a multi-agent orchestrator with named sub-agents and a failure dispatcher. The path between them is deliberate, and every step earns its place.

Course design principle: the Removal Test Every chapter is built so it collapses without Claude Desktop running it. Reading the Markdown alone gives you about 40% of the value. The other 60% comes from the dialog: the persona asking the calibration question, you predicting the code's output before running it, the persona pushing back when your answer is shallow. The course is genuinely AI-first — not a static syllabus with an AI bolted on.

👤 Who this is for

Right fit

  • You already understand AI fundamentals — you know what an LLM is, you've used one, you've read about agents.
  • You know the conceptual "five traits" of an agent (goal-orientation, perception, reasoning, action, iteration) and you want to learn how to incorporate them when designing your own.
  • You can write intermediate Python — async/await is familiar, you can read a 200-line file and reason about it.
  • You want hands-on architecture skill, not theory. You'd rather build a working orchestrator than read another paper.
  • You have a Claude Max subscription (or are willing to use the SDK with an API key — but the course is designed around Max).

Not the right fit (yet)

  • You're brand new to LLMs and want a "what is AI" overview. Start with a survey course first; come back here once you've used an LLM in anger.
  • You want a no-code or low-code agent builder. This course is Python and SDK primitives end to end.
  • You want JavaScript or TypeScript. The SDK supports both, but this course teaches the Python flavor.

🧱 How it's designed

The course rests on three architectural choices that distinguish it from a typical tutorial series.

1. Lego-block progression

Each chapter adds exactly one new thing. Chapter 1 is a bare LLM call so you can feel what an agent is not. Chapter 2 adds a goal predicate and nothing else. Chapter 3 adds composition. Chapter 4 adds iteration. By Chapter 8 you've integrated all five traits in one runnable file. By Chapter 12 you have a multi-agent orchestrator. You never get a chapter that secretly bundles three concepts — and when complexity arrives, you can name every piece that brought you there.

2. Persona-led, AI-first teaching

Each chapter has a named instructor persona — The Reductionist, The Cybernetician, The Conductor, and so on. Their voices are different by design. When you open a chapter in Claude Desktop and say "run me through this," Claude adopts the persona's voice and runs a five-phase loop: Calibration (a Socratic question to gauge where you are) → Adapt (shallow / strong / reinforce branches) → Code (predict first, then run) → Assessment (one mechanical question, scored) → Wrap (a one-line carry-forward insight).

If you give a shallow first answer, the persona drops to first principles. If you give a strong answer, it jumps to the nuance. If your prior chapter ended with a weak score on a specific concept, the next chapter opens with reinforcement on exactly that concept — a mechanism called weakSignal handoff.

3. Reading-fluency is not writing-fluency

After Chapter 12, the course doesn't stop at "you've seen the patterns." Chapter 12.5 is a 30-minute Cold Build: blank IntelliJ file, one-line spec, no looking at prior chapters. You either build a working agent in 30 minutes or you don't. The discomfort is the entire point. Then Chapter 13 covers what production actually needs — external MCP servers, retry/backoff, prompt-injection defense.

🎭 Meet your instructors

Fourteen named personas, one per chapter. Their voices are different by design — that's how you remember who taught you what. Each adopts the same five-phase teaching loop but with their own register and analogies. Below are the opening lines you'll actually hear, verbatim from the personas file.

The Reductionist · Ch 1 · The Bare LLM Call

"Hold this image: a function. Six lines. Return-and-die. Now tell me what it is NOT, that an agent IS."

The Strategist · Ch 2 · LLM Call + Goal

"What does success mean here? Write the check, not the wish. Models will confidently miss; the verifier is your contract with reality."

The Cybernetician · Ch 4 · Iteration + Context

"The loop is the spine. Now show me the three places it can die — and the disease that creeps in even when it doesn't."

The Observer · Ch 5 · Perceive

"The agent doesn't see what's there — it sees what you described. Show me the description, and I'll predict the behavior."

The Planner · Ch 6 · Reason + Tool Design

"Schema is grammar. Loose grammar, loose thought. Required fields are how you compress reasoning quality into the contract."

The Reviewer · Ch 9 · Verifier / Critic

"Don't ask the model 'is your answer correct?' — that's confirmation, not verification. Give me a structurally different judge."

The Incident Commander · Ch 10 · Recovery + HITL

"Three failure responses, three seams. Premature retry is the rookie mistake. When in doubt, stop the loop and ask a person."

The Conductor · Ch 12 · Orchestrator + Sub-Agents

"Whom do you dispatch first, and why? Before any model call, you've already made five architectural decisions. Name them."

The Engineer · Ch 13 · Production Surfaces

"Your agent works on your laptop. Now it has to work on someone else's machine, with someone else's network, and adversaries reading its inputs."

💬 What a chapter actually feels like

Below is an excerpt from a real sandbox simulation of Chapter 4 (Iteration). The Cybernetician opens by reinforcing the prior chapter's weak signal — control-plane ownership — before asking the calibration question.

Professor (The Cybernetician) Before we open Chapter 4, one callback. In Chapter 3 your weak signal was naming where Python owns the control plane — you spoke about the chain "flowing," but never pointed at the line of code holding the baton. Fix that with me in one sentence: where exactly did the control plane live, and what would it have looked like for control to leave Python?
Student Honestly — control was kind of in the orchestration. The script sort of drove things forward.
Professor That's the fuzzy language I want to retire. "Sort of drove" is not an architecture. Point at the mechanism. Control was the literal Python for statement that decided which prompt fires next. The model produced text; Python decided what happens after. Control leaves Python the moment the model — not your code — decides whether to call again. Say it back cleanly.
Student Got it. Python's sequential calls — call_1() then call_2(output_of_1) — were the control plane. Control leaves Python when the loop's continuation condition is a function of the model's output that the model can influence.
Professor Now you're holding the baton. Chapter 4 is the exact moment it changes hands. Calibration question: you have a goal predicate from Ch 2 and chained calls from Ch 3. What's the minimum change that turns this into something you'd call an agent?

🛠️ What you'll be able to build by Chapter 13

Concretely — not aspirationally. After completing the course, you can sit down at IntelliJ and architect:

  • A single agent with a goal predicate, an iteration loop, and three exit conditions, where you can defend every design choice with the underlying tradeoff.
  • A tool surface for any task: read-only sensors with rich descriptions, write tools with permission gates, structured plan tools whose schema is the agent's grammar.
  • A failure dispatcher that routes errors to retry / re-plan / escalate, with budgets and promotion rules.
  • A verifier loop where the actor and judge are structurally separated.
  • A two-agent system with a contract over durable shared state — including restart resilience when one process dies mid-task.
  • A multi-agent orchestrator with three sub-agents at mixed model tiers (haiku for cheap reads, sonnet for generation and review), aggregating results.
  • An external MCP server connection, a network-backed tool with retry/backoff, and an injection-defended user-content handler.
  • An eval set you wrote yourself, scored deterministically or via LLM-judge, gating every change you ship.

💡 Tips for getting the most out of it

  1. Don't skip the predict step. Every chapter asks you to predict the code's output before running it. Skipping this is the single highest-leverage thing you can do wrong. Predicting forces the mental model you're here to build.
  2. Stick to one chapter per deep-work block. 60–90 minutes, undisturbed. Cramming two chapters in one sitting reliably dilutes the reflection — and the reflection is where the skill lands.
  3. Run the brief after every chapter. The cumulative HTML brief is your trajectory. Watching it grow over three weeks is the single best evidence you're learning.
  4. Be honest in the assessment. If the persona scores you below 0.7, don't argue — that triggers reinforcement next chapter. Lying to a Socratic tutor only hurts you.
  5. Do the Cold Build at 12.5 ungraded. The whole point is to surface what you still need to look up. Treat the lookup list as your re-study plan.
  6. Re-read the Chapter 12 capstone checklist before any real project. Nine items. Print it. Tape it next to your monitor.
  7. If you get stuck, look at the persona's voice opener. It's a one-sentence reminder of the chapter's whole point — calibration anchor for when you've drifted.

❓ Frequently asked

Do I need an API key?
No. The Claude Agent SDK can use the Claude Code CLI's stored OAuth session. claude login once with your Max account and you're set. Make sure ANTHROPIC_API_KEY is NOT in your IntelliJ run config — it would silently bill the API instead.
How long does the course take?
About 16 hours of focused work, ideally spread across three weeks at one chapter per session. You can compress it; you'll lose depth if you do.
What if I'm a brand-new Python developer?
The course assumes intermediate Python — you can read async/await, you can reason about a 200-line file. If that's not you yet, build that muscle first; the course will be more valuable a month later.
Does it work without Claude Desktop?
The code files run fine standalone. But the chapter Markdowns are designed for Claude Desktop to read and run with you. Without the dialog, you get ~40% of the value (that's the Removal Test working as designed).
Can I share my session briefs?
Yes — they're self-contained HTML files. They're your portfolio of what you've learned. Several students have used them in interview portfolios.
What's the Cold Build chapter actually like?
Blank IntelliJ. One-line spec: "build an agent that scans Markdown files for broken internal links and proposes fixes." Thirty minutes on a timer. No looking at prior chapters. You either build a working agent or you don't. The discomfort is the point — it's the only chapter that measures writing-fluency vs reading-fluency.
What if I fail the Cold Build?
Look at what you had to look up. Each item is a re-study target. Do the same exercise next month with a different spec. Repeat until 30 minutes feels like enough.

⚠️ Honest limitations

Three things this course doesn't do, named upfront so you can decide if they matter to you.

  • No TypeScript flavor. The Claude Agent SDK supports both Python and TypeScript. This course teaches Python only. If you need TypeScript, the architectural concepts translate but the code does not.
  • Deployment is a single chapter, not a thread. Chapter 13 covers production surfaces (external MCP, network resilience, injection defense). It doesn't cover containerization, runtime topology, auth flows, or model migration. That's a separate course.
  • The personas are designed for an architect-grade learner. They don't perform agreement; they push back. If you want a warm, encouraging tutor that always validates you, this isn't it. The personas are direct because the work is direct.
